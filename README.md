# Polyformalism: Multi-Formalism Creative Cognition

**Think in structures that aren't native to you. Solve problems by imposing constraints from outside your default cognitive mode.**

Polyformalism is the practice of solving a problem in one formalism by forcing it through the constraints of another formalism — writing code as prose, designing architecture as music, analyzing data as geometry. When your native formalism produces the same dead-end answer, a different formalism can break the loop.

---

## The Thesis

Every formalism — programming language, mathematical structure, natural language, musical notation — imposes a constraint system on thought. These constraints aren't weaknesses. They're levers.

A problem that has no solution in linear algebra might have a clean solution in graph theory. A system design that's impossible in Rust's ownership model might be natural in Erlang's actor model. The formalism IS the solution space. Changing the formalism changes what's possible.

Polyformalism is the systematic practice of exploiting this.

---

## The 7-Type Taxonomy

Our empirical work identified 7 distinct polyformalism types:

| Type | Move | Example |
|------|------|---------|
| **Translation** | Map between formalisms with the same semantics | Rust → C → WASM |
| **Analogy** | Transfer structure across domains | Circuit design → social network topology |
| **Constraint injection** | Force-propagate one formalism's rules through another | Type system constraints into data validation |
| **Hybridization** | Combine formalisms into a new one | Sheaf theory × distributed consensus |
| **Inversion** | Solve the dual problem | Instead of proving correctness, prove impossibility of incorrectness |
| **Vacillation** | Alternate between formalisms until convergence | Architect → prototype → test → redesign |
| **Metamorphosis** | Let a solution change formalisms mid-stream | An algorithm that starts as math, compiles to hardware, and deploys as firmware |
